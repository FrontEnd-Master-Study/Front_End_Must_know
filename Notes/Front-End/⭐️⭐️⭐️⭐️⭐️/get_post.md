## Http 메소드

**_클라이언트가 웹 서버에게 사용자 요청의 목적이나 종류를 알리는 수단이다_**

초기에는 GET만 있었으며 점차 늘어나서 지금은 9가지가 있고 이중 5가지가 주로 쓰이고 있다.

<br>

### Http 메소드의 속성

- 1. 안전 : 계속해서 메소드를 호출해도 리소스를 변경하지 않음.
- 2. 멱등 : 메소드를 계속 호출해도 결과가 똑같음.
- 3. 캐시 가능 : 캐싱을 해서 데이터를 효율적으로 가져올 수 있음.

<br>

### 주요 메소드 5가지

- GET : 리소스 조회
- POST : 요청 데이터 처리, 주로 데이터 등록에 사용
- PUT : 리소스를 대체, 해당 리소스가 없으면 생성
- PATCH : 리소스를 일부만 변경
- DELETE : 리소스 삭제

<br>

## GET

```
GET 메서드는 특정 리소스의 표시를 요청한다. 오직 데이터를 받기만 하는데 사용
```

_데이터를 읽거나(Read), 검색(Retrieve)할 때에 사용되는 메소드_

<br>

### 사용 방법

<br>

URL 주소 끝에 파라미터로 포함되어 전송되며, 이 부분을 쿼리 스트링(QueryString) 이라고 부릅니다.

파라미터가 여러 개이면 `&`로 연결합니다.

> `e.g.) www.example-url.com/resources?name1=아이언맨&name2=스파이더맨`
>
> > 파라미터인 name1과 name2를 통해 값을 전달받을 수 있습니다.

<br>

### GET의 특징

- GET은 불필요한 요청을 제한하기 위해 요청이 캐시될 수 있습니다.
- 파라미터에 내용이 노출되기 때문에 민감한 데이터를 다룰 때 GET 요청을 사용해서는 안 됨.
- GET 요청은 브라우저 기록에 남는다.
- GET 요청을 북마크에 추가할 수 있음.
- GET 요청에는 데이터 길이에 대한 제한이 있음.
- Get 요청은 성공시, 200(Ok) HTTP 응답 코드를 XML, JSON뿐만 아니라 여러 데이터(html, txt등..), 여러 형식의 데이터와 함께 반환함.
- GET 요청은 **멱등**함.

<br>

## POST

```
POST 메소드는 리소스를 생성/업데이트하기 위해 서버에 데이터를 보내는 데 사용한다.
```

_리소스를 생성하거나 업데이트할 때 사용하는 메소드_

<br>

### POST의 특징

- GET과 달리 전송해야할 데이터를 HTTP 메세지의 Body에 담아서 전송.
- GET과 달리 대용량 데이터를 전송할 수 있음 (HTTP 메세지의 Body는 길이 제한 없이 데이터 전송 가능)
- POST 요청은 캐시되지 않음.
- POST 요청은 브라우저 기록에 남아 있지 않음.
- POST 요청 중 자원 생성은 201(Created) HTTP 응답 코드를 반환.
- POST 요청은 **멱등하지 않음**

<br>

💡 POST 요청은 데이터가 Body에 전송되고,눈에 보이지 않아 GET보다 안전하다고 생각할 수 있지만,여러 툴 들로 요청 내용을 확인할 수 있기 때문에 **민감한 데이터의 경우 반드시 암호화해 전송해야함**

## GET vs POST

<img width="450" alt="스크린샷 2022-03-31 오후 6 56 35" src="https://user-images.githubusercontent.com/48541850/161029209-49190a4f-87b5-4dbe-af80-53ac3affbbb1.png">

<br>

|                  | GET             | POST            |
| ---------------- | --------------- | --------------- |
| 캐시             | ⭕️             | ❌              |
| 브라우저 기록    | ⭕️             | ❌              |
| 북마크 추가      | ⭕️             | ❌              |
| 데이터 길이 제한 | ⭕️             | ❌              |
| HTTP 응답 코드   | 200(ok)         | 201(Created)    |
| 주로 사용되는 때 | **리소스 요청** | **리소스 생성** |
| 리소스 전달 방식 | 쿼리 스트링     | HTTP Body       |
| 멱등성           | ⭕️             | ❌              |

<br>

_GET은 **Idempotent**(멱등)하게 설계되었고 POST는 **Non-Idempotent**하지 않게 설계되었음_

## PUT

목적 리소스를 현재 메세지의 값으로 생성하거나 만약 존재한다면 기존 리소스를 삭제하고 **덮어쓰기** 한다

> **💡 POST와 PUT의 구분** <br>
> PUT은 POST와 다르게 클라이언트가 리소스의 위치를 알고 URI를 지정해 주어야 한다 <br>
> ex) POST /members <br>
> ex ) PUT /members/100

## 요약

### HTTP 메소드 5가지

- GET : 리소스 검색에 사용
- POST : 주로 리소스 생성에 사용
- PUT : 리소스 수정에 사용
- PATCH : 리소스 수정에 사용, (리소스를 일부분만 변경 가능)
- DELETE : 리소스 삭제에 사용

<br>

### GET과 POST 차이

|                  | GET             | POST            |
| ---------------- | --------------- | --------------- |
| 캐시             | ⭕️             | ❌              |
| 브라우저 기록    | ⭕️             | ❌              |
| 북마크 추가      | ⭕️             | ❌              |
| 데이터 길이 제한 | ⭕️             | ❌              |
| HTTP 응답 코드   | 200(ok)         | 201(Created)    |
| 주로 사용되는 때 | **리소스 요청** | **리소스 생성** |
| 리소스 전달 방식 | 쿼리 스트링     | HTTP Body       |
| 멱등성           | ⭕️             | ❌              |

<br>

### HTTP 상태 코드

- 1xx (Informational): 요청이 수신되어 처리중
- 2xx (Successful): 요청 정상 처리
- 3xx (Redirection): 요청을 완료하려면 추가 행동이 필요
- 4xx (Client Error): 클라이언트 오류, 잘못된 문법등으로 서버가 요청을 수행할 수 없음
- 5xx (Server Error): 서버 오류, 서버가 정상 요청을 처리하지 못함
